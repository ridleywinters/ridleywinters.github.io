#### What is Sea? 

Sea is an **extension of Markdown** designed for **data-driven organization** that can be used to build dynamic web sites and knowledge. Sea has the **fluency of Markdown** combined with the data-driven extensibility of **React** and **YAML**.  It is designed for **incremental, progressive improvement** from idea to production-quality work. Think of it as an **agile content and presentation** language.  It intentionally tries to be as **minimal** as possible while acheiving these goals.

There's one basic tool:

```bash
$ sea build dirname/  --out database.json # builds database.json
```

Then to host a site all that is needed is to copy `database.json` into a folder with a a base `index.html` and `sea-site.js`.  Site customization exists entirely in the database itself, so there's no need to modify the host html or script.  Note: this is a _single page static website_ so free hosts like GitHub Pages can easily host this. No operational burden!

For active development, to launch a local server with hot reloading:

```bash
$ sea-build dev .
```

How do I customize it? If you know basic JavaScript and React, it should be very easy: Sea tries to inject as few "framework specific" concepts as possible.


Note: Sea is more of a _technology_ than it is a _product_. It's very flexible and customizable. Sea templates can overlay an opinion on the technology, creating a clear, immediately usable use cases - thus making Sea into more of a product.

Can be thought of as a programming language.

## What problems does it solve?

Semantic wiki, TiddlyWiki, MediaWiki, Markdown, Gatsby, database, spreadsheets, etc. Another swing at a balance of that functionality.  Incremental development of ideas.  Allow things to be "finished" and published sooner yet continually improved.

## Quick Start

Install the `sea` command-line program:

```bash
$ npm install --global @ridleywinters/sea
```

Create a barebones example project (note: if you skip the `--template` argument, a bare-minimal project will be created):

```console
$ sea create myproject --template=example
```

You should now have a directory structure that looks something like this:

```
myproject\
    pages\
        home.mdx
        page1.mdx
```

Now run the development server:

```
$ sea dev
```
Lastly:

* Visit http://localhost:8080!
* Start modifying the MDX files to see the changes livee
* Check out the [features](#features) section to play with some of the more interesting and unique features of Sea


## How does it work?

Sea operates on a directory of files, typically stored in git (for personal knowledge bases) or in a database for large multi-user use cases. Each file represents a **Card** - a sttarting point, think of that as a fairly standard Markdown document. Sea parses the whole set of Cards and creates a tree of data with indices so that a renderer can then easily render the documents as well cross-reference information in other Cards.

## How does it compare?

Sea is an extension of Markdown into multiple files to create a customizable database of information.  It has the fluency of Markdown for simply writing text with minimal markup to support specifying data and formatting. It is customizable in that additional data types and renderers can be added to the database _directly_ as part of the database itself (no plug-ins, installation config, etc.).

## Musings

In one sense, Sea is a literal programming language for iteratively moving from idea to implementation. It's one that "starts" with plain text. Markdown is then layered in to a structure to the text.  Data objects and transformations can then be added in to programmatically turn inputs into new outputs. Exporters can then make these transformations available in external tools (whether that's a binary asset, a code package, etc.).

### Custom nodes

The following creates a custom node in the AST of type `user_story`. The contents of the fenced section is parsed as YAML.

~~~
```@user_story
story: As a user of this application, I want to specify custom objects in YAML so I can implement more visually appealing custom rendering. 
acceptance_criteria:
    - Must work
    - Must work well
```    
~~~

By default, the renderer looks for a React renderer of the same name (in this case, `user_story.jsx`) and uses that to display the data object:

```@user_story
story: As a user of this application, I want to specify custom objects in YAML so I can implement more visually appealing custom rendering. 
acceptance_criteria:
    - Must work
    - Must work well
```

### Embedding specific data types

This example uses the `$<type>` suffix on a YAML field to indicate the parser that should be used on that field. In this case, it ensure the YAML field is parsed as Markdown (`$md`) so the `content` field is an AST as well.

~~~
```@expand
title: This is an Expand object
content$md: |
    This is a longer bit of content.  I would prefer if this supported markdown
    and multiple lines, though. I need to consider some way to keep the AST
    going.

    * List Item A
        * Sub-item A1
        * Sub-item A2
    * List Item B
    * List Item C
```
~~~
