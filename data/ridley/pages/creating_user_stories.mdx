# Product Development through User Stories

_There are many approaches to product development, this is just one possible take on it..._

## Goal

The goal of this **user story centered** approach is to create a lightweight approach that both helps guide effective day-to-day work but also covers strategic, year-long planning in a connected way.  

## Overview

This approach puts **[[user stories]]** at the forefront of what is generated and iterated on. These are collected into fixed size **epics** so a timeline can be formed to allow for stakeholders planning around the projects.  As these initially evolve a short **project charter** is created to set a long-term north-star for the team. These work together to create a balance of customer value deliverability, predictiability, and iteration on what is learned along the way.

This approach has a bias towards scope-reduction over schedule or resourcing adjustment (in the traditional scoped-schedule-resources balance of product development).

## Getting started

This section assumes you're starting from scratch on this process. After working in this fashion for a while, it becomes a more natural, iterative processs of improving on the prior work: we'll get to that in the next section.

Assuming you're working with great people, they're probably not excited about learning a new process, but rather getting start to doing higher-quality work: this section tries to speed start to the value and layer in the process that matters as it goes.

### Generate a set of user stories

Expect the user stories to be varied and disorganized. The team is not yet aligned, so that's to be expected. Remember the following:

* Anyone can write a story
* The story should fit the format: _As a {user role}, I want to {goal or desire} so that {why}._
* During brainstorming, do allow and record _I want to {goal}_: this can be refined later. Capture what people care about first.
* These can be refined, grouped, and prioritized over time, but all are valid

Pair each with an **acceptance criteria**, which is a set of 1-3 unambiguous questions that, when all can be answered yes, the work can be considered done. This is also called the "definition of done."  This piece is critical to enable people to scope work back to the minimal viable solution (and thus meet a value-delivery timeline).

Aside: functional stories are not often part of Agile process. The reason being they don't deliver direct customer value, which is supposed to be the entire intent of product development, right? Having worked as a product manager at a highly technical company (where much of the company value was based on some expert-class engineering), I found ignoring the functional stories can remove engineers excellent skill in identifying opportunities, economic advantages. This ignores a major value add.

Try not confuse user stories or functional stories with tasks. If you have tasks that feel exciting, write them down but hold off on bringing them up until the planning phase.


### Refining user stories

This isn't a discrete step, but rather should happen as naturally as possible as part of either the generation or review of user stories.  User stories should not aim for perfection, but do look for ways to improve them:

* Is the goal focused on the user value? If not, can the story be rewritten in a way that expressed why the user cares about this?
* Does the story emphasize how this needs to be implemented? If yes, can that be removed and still express the user value?  If yes _but there's a functional requirement that really matters embedded there_, move that to the acceptance critiera.

## FAQ

### How do I keep the product cohesive? Everyone on my team has good ideas but they're all so different!

One approach - usually not best when first getting started - is bridging user stories before beginning work on them. Combine a succesffully completed user story and the new story:

_As a {user rule}, I use {product name} to {prior want} so {goal/why}.  I {new want} so that {goal/why}._

This can be helpful when existing user flows in the product may need attention due to their being incomplete, problematic for users, or there're opportunities to get more value from it. Generally if using this approach do not impose that the team must use a specific prior user story during the brainstorming process: this will stifle creativity. Instead rely on the charter to guide attention on the right areas.

### Why use User Roles rather than personas?

User Roles intend to places more emphasis on the activities the person wants or needs to do. This avoids getting caught up in the job titles (which do not always map to the same responsibilities across companies) or irrelevant individual demographics biases.
